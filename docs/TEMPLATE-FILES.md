## üìÅ Regular Project Template Files

### Configuration Files

#### `.gitignore`

**Purpose:** Specifies files and directories that Git should ignore

**Includes:**

- OS-generated files (.DS_Store, Thumbs.db)
- Editor files (.vscode/, .idea/, \*.swp)
- Logs and temporary files
- Backend dependencies (node_modules/, venv/, **pycache**)
- Database files (_.db, _.sqlite, instance/\*.db)
- Frontend build outputs (dist/, build/, .next/)
- Environment variables (.env, .env.local)
- Security files (_.pem, _.key, secrets/)

**Customization:**

- Add project-specific patterns
- Modify database exclusions based on your setup
- Update for specific frameworks or tools

#### `.dockerignore`

**Purpose:** Specifies files to exclude from Docker build context

**Includes:**

- Version control (.git/)
- Documentation (\*.md, docs/, admin/)
- CI/CD files (.github/)
- Editor files (.vscode/, .idea/)
- Testing files (tests/, coverage/)
- Development dependencies (node_modules/, venv/)
- Database files (\*.db, instance/)
- Environment files (.env)

**Customization:**

- Adjust based on Docker build requirements
- Keep necessary runtime dependencies
- Consider multi-stage build needs

### Project Files

#### `README.md`

**Purpose:** Main project documentation and entry point

**Includes:**

- Project overview and purpose
- Quick start guide
- Technology stack
- Development workflow
- Project structure
- Links to detailed documentation

#### `start.txt`

**Purpose:** Project initialization template

**Includes:**

- Problem statement
- Project type selection
- Scope definition
- Structure preferences
- Version control options
- Tool integrations
- CI/CD preferences

#### `.github/workflows/ci.yml`

**Purpose:** CI/CD automation configuration

**Includes:**

- Branch-based workflow detection
- Automated testing and linting
- Build and deployment
- External review integration

## üîß File Management

### Generated by new-project.sh

The project generator script (`scripts/new-project.sh`) automatically:

1. **Copies all template files** including hidden files (.gitignore, .dockerignore)
2. **Verifies important files** exist after copying
3. **Customizes files** with project-specific information
4. **Initializes git repository** with appropriate ignores

### Verification Process

The script checks for:

- `.gitignore` presence (warns if missing)
- Template directory existence
- Successful file copying

### Customization Process

After generation, the script:

- Updates `start.txt` with project details
- Customizes `README.md` with project name and description
- Preserves all ignore files as-is

## üé® Customization Guidelines

### .gitignore Customization

**Add patterns for:**

- Framework-specific files
- Build artifacts
- Generated code
- API keys and secrets
- Large binary files

**Example additions:**

```gitignore
# Framework specific
.next/          # Next.js
.nuxt/          # Nuxt.js
target/         # Maven/Gradle

# IDE specific
*.iml           # IntelliJ
.classpath      # Eclipse

# Project specific
/generated/
/cache/
```

### .dockerignore Customization

**Consider:**

- Multi-stage build requirements
- Runtime vs build dependencies
- Image size optimization
- Security considerations

**Example additions:**

```dockerignore
# Keep production dependencies
!package-lock.json
!requirements.txt

# Exclude development tools
Makefile
docker-compose.dev.yml
```

## üö® Important Notes

### Hidden Files

- `.gitignore` and `.dockerignore` are hidden files (start with `.`)
- Use `ls -la` to verify they exist in your project
- Most IDEs show hidden files by default
- Git will track these files (they're not ignored by themselves)

### Template Updates

When templates are updated:

- New projects get latest versions automatically
- Existing projects need manual updates
- Compare your files with latest templates
- Merge customizations carefully

### Common Issues

**Missing .gitignore:**

- Verify template has the file
- Check copy command preserves hidden files
- Use `cp -r` to ensure all files copied

**Accidental commits:**

- Review .gitignore before first commit
- Use `git status` to check tracked files
- Add forgotten patterns immediately
- Use `git rm --cached` to untrack files
